# MySQL 월별 파티셔닝 — 근태 대용량 테이블 설계

> 전 사원 × 매일 누적되는 `commute_record` / `attendance` 테이블을 **`RANGE COLUMNS(work_date)` 월별 파티셔닝**으로 설계한 의사결정 과정과, 운영 중 마주친 JPA 더티체킹 함정을 정리합니다.

[← README로 돌아가기](../README.md)

## 1. 파티션 테이블이란

**파티셔닝(Partitioning)** — 논리적으로는 하나의 테이블, 물리적으로는 기준에 따라 분할 저장하는 기법.

- 애플리케이션: 단일 테이블 (`SELECT * FROM commute_record`)
- MySQL 내부: 파티션별 `.ibd` 파일 분리
- 파티션 키 조건이 WHERE 에 있으면 **해당 파티션만 스캔** (Partition Pruning)

### 분할 기준 (파티션 종류)

| 방식 | 기준 | 예 |
|------|------|----|
| **RANGE** | 값의 범위 | 월별 / 연도별 분할 |
| **LIST** | 명시된 값 목록 | 지역코드별 분할 |
| **HASH** | 해시 함수 결과 | 사원 ID 기준 균등 분산 |
| **KEY** | MySQL 내부 해시 | PK 기준 균등 분산 |

근태 테이블은 **시간순 누적 + 월 범위 조회 위주** 라 `RANGE(work_date)` 가 최적.

## 2. 인덱스와 뭐가 다른가

둘 다 "조회 성능을 높인다" 는 점에서 자주 혼동되지만, **동작 방식이 근본적으로 다릅니다.**

| 항목 | 인덱스 | 파티셔닝 |
|------|--------|----------|
| 목적 | 행을 빠르게 **찾기** | 데이터 자체를 **물리적으로 분리** |
| 데이터 위치 | 한 테이블 안에 모든 행이 섞여 있음 | 기준값에 따라 별도 파티션에 격리 |
| 자료 구조 | B-Tree (별도 인덱스 파일) | 파티션 = 독립된 저장 단위 |
| 조회 방식 | 인덱스 트리 탐색으로 행 위치 획득 | 불필요한 파티션을 스캔 대상에서 제외 |
| 데이터 누적 시 | 트리 깊이 ↑ → 탐색 비용 ↑ | 파티션 단위 크기는 고정 → 비용 일정 |
| 오래된 데이터 삭제 | `DELETE` → 행 단위, 락·binlog 부담 | `DROP PARTITION` → 메타데이터만, 거의 무비용 |
| 백업 / 아카이브 | 테이블 전체 단위 | 파티션 단위 분리 가능 |

### 한 줄 요약

> **인덱스** — 어디 있는지 빠르게 찾자
> **파티셔닝** — 애초에 안 봐도 되는 데이터는 건드리지도 말자

둘은 **경쟁이 아니라 보완 관계** — 파티션으로 범위를 좁힌 뒤 그 안에서 인덱스로 행을 찾는 구조가 가장 효율적.

## 3. 왜 파티셔닝을 사용해야 했는가

### 근태 도메인 특성
- `commute_record`, `attendance` 는 **전 사원 × 매일** 누적 → 수년 내 수천만 행 적재
- 조회 패턴의 90% 이상이 **월 범위(`work_date BETWEEN`)** 조건 (월간 근태 집계, 급여 산정)
- 단일 테이블 + 인덱스만으로는 누적될수록 B-Tree 깊이가 깊어져 조회 비용이 **시간 흐름에 따라 선형 누적**

### 검토한 대안

| 옵션 | 장점 | 단점 | 채택 |
|------|------|------|------|
| 단일 테이블 + 인덱스 강화 | 단순 | 누적 데이터에 한계, 핫/콜드 분리 불가 | ❌ |
| HASH 파티셔닝 (`empId` 기준) | 사원별 분산 | **월 범위 조건 시 전 파티션 풀스캔**, 동적 삭제 불가 | ❌ |
| 애플리케이션 레벨 샤딩 (월별 테이블 분리) | 프루닝 효과 동일 | JPA·QueryDSL 매핑 복잡도 폭증 | ❌ |
| `RANGE COLUMNS(work_date)` 월별 | 월 범위 자동 프루닝, 핫/콜드 분리 가능 | 복합 PK 필요 | ✅ |

### 최종 설계
- `RANGE COLUMNS(work_date)` **월별 파티셔닝**
- 복합 PK `(com_rec_id, work_date)` — MySQL 제약상 파티션 키는 PK 의 일부여야 함
- `PartitionScheduler` 가 매월 **다음 달 파티션을 선행 생성** → 월 경계 INSERT 실패 방지

## 4. 성능이 얼마나 좋아지는가

### 핵심 시나리오 — 월간 근태 집계

```sql
SELECT * FROM commute_record
 WHERE com_id = ?
   AND work_date BETWEEN '2026-04-01' AND '2026-04-30';
```

| 항목 | Before (단일 테이블) | After (월별 파티셔닝) |
|------|---------------------|-----------------------|
| 스캔 대상 | 전체 누적 행 (수천만) | 4월 파티션만 (수십만) |
| 스캔 비율 | 100% | **약 1/N** (N = 누적 개월 수) |
| 인덱스 깊이 | 누적될수록 깊어짐 | 파티션 단위로 고정 |
| 응답 시간 | 적재량에 비례해 우상향 | 적재량과 무관하게 **거의 일정** |

> 운영 3년 시 약 1/36 수준만 스캔. 데이터가 쌓일수록 격차는 더 벌어짐.

### 운영 작업

| 작업 | Before | After |
|------|--------|-------|
| 오래된 데이터 삭제 | `DELETE WHERE work_date < ...` → 수십만 행 락·binlog 폭증 | `ALTER TABLE ... DROP PARTITION` → 메타데이터만 갱신, 즉시 완료 |
| 백업 / 아카이브 | 테이블 전체 | 파티션 단위 분리 가능 |
| 핫/콜드 분리 | 불가 | 오래된 파티션은 별도 스토리지로 detach 가능 |

## 5. 사용 시 발생할 수 있는 문제

파티셔닝은 강력한 만큼 **잘못 쓰면 효과가 그대로 사라집니다.** 실제 운영 중 마주친 함정.

### 5-1. 함정 — JPA 더티체킹이 파티션을 무력화

> **전제** — 파티션 프루닝은 WHERE 절에 파티션 키가 있어야 작동. 없으면 옵티마이저가 어느 파티션을 봐야 할지 알 수 없어 **전 파티션 풀스캔** → 단일 테이블보다 오히려 느릴 수 있음. 더티체킹은 이 전제를 가장 흔하게 깨뜨림.

**문제 사항**
- 근태 정정 승인·체크아웃·자동마감 등 분 컬럼 업데이트 시 **DB CPU 90%+** 로 튐
- 월별 파티셔닝을 적용했음에도 효과가 사실상 무력화

**원인 분석**
- 복합 PK 중 파티션 키(`work_date`)가 JPA 매핑에서 빠져 있음 (단일 `id` 만 매핑)
- 더티체킹이 생성하는 UPDATE 문에 파티션 키 누락 → `WHERE com_rec_id = ?` 만 들어감
- WHERE 에 파티션 키 없으면 옵티마이저가 파티션을 좁히지 못해 **모든 월별 파티션 풀스캔** → CPU 폭증

**시도 방법**

> 방향성 — **매핑 구조는 그대로 두고 갱신 경로만 손대는** 쪽으로 옵션을 좁힘 (회귀 영향 최소화).

| 옵션 | 결과 | 채택 |
|------|------|------|
| 더티체킹 유지 + 인덱스 보강 | 인덱스로는 파티션 프루닝을 살릴 수 없음 | ❌ |
| `@IdClass` / `@EmbeddedId` 로 복합 PK 매핑 | 다른 모듈 연관관계 매핑 연쇄 변경 + 코드 복잡도 폭증 | ❌ |
| native UPDATE 일원화 + WHERE 에 파티션 키 명시 | 매핑 변경 없이 프루닝 강제 | ✅ |

**해결 방법**
- 분 컬럼 갱신을 **native UPDATE 로 일원화** → WHERE 에 파티션 키 명시로 프루닝 강제
- `@Modifying(clearAutomatically = true)` + 후속 로직 전 엔티티 재로드로 영속성 컨텍스트 동기화
- 영향행 수 검증으로 atomic check-and-update 보장

```java
@Modifying(clearAutomatically = true)
@Query(value = """
    UPDATE commute_record
       SET check_out_time = :checkOutTime,
           work_minutes   = :workMinutes
     WHERE com_rec_id = :id
       AND work_date  = :workDate     -- ← 파티션 프루닝 강제
    """, nativeQuery = true)
int updateCheckOut(...);
```

### 5-2. 그 외 주의사항

| 문제 | 설명 | 대응 |
|------|------|------|
| 파티션 자동 증설 누락 | 다음 달 파티션을 안 만들어두면 월 경계 INSERT 실패 | `PartitionScheduler` 로 선행 생성 |
| 복합 PK 강제 | MySQL 은 파티션 키가 PK/Unique Key 일부여야 함 | `(id, work_date)` 복합 PK 채택 |
| 파티션 간 쿼리 | 여러 파티션을 동시에 스캔하면 효과 감소 | 조회 범위를 월 단위로 제한 |
| 잘못된 파티션 키 선택 | 조회 조건에 거의 안 들어가는 컬럼으로 분할하면 의미 없음 | 도메인 조회 패턴 분석 후 선택 |

## 6. 운영 규칙

| 규칙 | 이유 |
|------|------|
| `commute_record` UPDATE 는 **반드시 native + `work_date` 포함** | 더티체킹 = 파티션 풀스캔 |
| 새로운 분 컬럼 추가 시 동일 패턴 적용 | 같은 함정 반복 방지 |
| `PartitionScheduler` 는 다음 달 파티션을 **선행 생성** | 월 경계 INSERT 실패 방지 |
| 오래된 월은 파티션 단위 detach/archive | 인덱스 깊이 유지 |

## 7. 결론

- 파티셔닝 = **"누적 데이터를 안 봐도 되게 만드는"** 기법, 인덱스(빨리 찾기) 와는 보완 관계. 근태처럼 **시간순 누적 + 범위 조회** 패턴에 최적이며 운영 N 개월 시 스캔량을 약 **1/N** 수준으로 축소.
- 효과는 **WHERE 절 파티션 키 강제** 전제 위에 성립. JPA 더티체킹은 이를 가장 흔하게 깨뜨리는 함정 → **native UPDATE 일원화** 로 차단.
- 매핑 구조는 단일 `id` 로 유지(다른 모듈 회귀 0), **갱신 경로만 native 로 통제** 하는 절충이 운영상 최적해.
